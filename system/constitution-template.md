## Architecture Principles

### 1. Simplicity First
- Choose the simplest solution that satisfies the requirements
- Avoid over-engineering: don't build for imagined scale that doesn't exist yet
- Prefer established patterns over clever abstractions

### 2. Separation of Concerns
- UI components don't contain business logic
- Business logic doesn't contain infrastructure concerns (DB, APIs)
- Infrastructure adapters are isolated from domain logic

### 3. Explicit Over Implicit
- No magic — configuration is visible and readable
- Dependencies are explicit in function signatures
- Side effects are isolated and documented

### 4. Fail Fast
- Validate inputs at system boundaries
- Return errors early, don't propagate invalid state
- Use TypeScript strict mode / strong typing always

## Testing Standards

### What to Test
- **Unit tests:** All pure business logic functions
- **Integration tests:** All API endpoints and database interactions
- **E2E tests:** All primary user flows
- **Visual tests:** All screens with defined UI specifications

### How to Test
- Test files live next to the source they test, following the project's stack convention
- E2E tests live in a dedicated directory (e.g. `tests/e2e/`)
- Visual baselines live in `.specs/features/NNN/baselines/`
- Every test description references the AC it validates

### Visual Testing Rules
- Visual baselines (via the project's visual testing tool) are captured on first successful implementation
- If a visual test diff exceeds 2% threshold → test FAILS
- Baselines are archived (not deleted) when spec changes
- Visual tests run in CI on every PR

### Testing Reference
- See `.specs/testing/strategy.md` for the full project testing strategy
- Concrete test commands are resolved in `.specs/testing/strategy.md` (Resolved Test Commands table)

## Code Conventions

### Naming
- **Files:** kebab-case (`user-notifications.ts`)
- **Components:** PascalCase (`NotificationBell.tsx`)
- **Functions:** camelCase (`getUserNotifications()`)
- **Constants:** SCREAMING_SNAKE_CASE (`MAX_NOTIFICATION_COUNT`)
- **Database tables:** snake_case (`user_notifications`)

### Structure
- Max file length: 300 lines (split beyond that)
- Max function length: 50 lines (extract helpers beyond that)
- No God objects / God components — single responsibility

### Formatting
- Prettier for formatting (config in `.prettierrc`)
- ESLint for linting (config in `.eslintrc`)
- No unused imports, no commented-out code committed

## Spec Conventions

### Mermaid Required
- Every user story in a spec.md MUST have a Mermaid flowchart
- Every plan.md with API calls MUST have sequence diagrams
- Diagrams are not optional — they are part of the spec

### Living Documentation
- Specs are updated when behavior changes — never let them become stale
- implementation.md is updated after every code change, with `@spec` anchor comments placed in source files to maintain resilient spec-to-code traceability
- changelog.md gets an entry for every feature/bugfix/refactor

### Update Rules
- **Behavior change:** Update spec.md + implementation.md + changelog.md
- **Code refactor (no behavior change):** Update implementation.md + changelog.md only
- **Bug fix:** Update implementation.md + changelog.md with Bugfix type entry
- **Spec update only:** Update spec.md + changelog.md with Spec Update type entry

## Decision Rules

When uncertain about an architectural choice, apply these rules in order:

1. **Does the constitution say anything about it?** → Follow it
2. **Does an existing pattern in the codebase solve it?** → Follow the pattern
3. **Is there a relevant ADR in `.specs/stacks/decisions/`?** → Follow the decision
4. **Still unclear?** → Add a `[DECISION NEEDED]` marker and ask the human

Do NOT make silent architectural decisions. Document them.
